Java与JVM：（四）并发的基石——CAS操作

发表于 2018-02-20 | 分类于 Java

背景

Synchronized是众所周知的锁策略，而Synchronized是一种悲观锁策略，在吞吐量上可能会有局限性（特别是早期的Synchronized，JDK1.6的偏向锁和轻量级锁为Synchronized带来了巨大的改进，这些更新同样依赖于CAS）。乐观锁则一般利用CAS来实现，在Java中位于java.util.concurrent包中，CAS可以说是撑起了一个concurrent包，足见其重要性。

CAS的全称概括了它的过程——Compare And Swap，整个操作主要包括三个值：当前内存某位置的值V，对V的预期值A，对V的欲更新值B。整个过程可以概括为“我认为V的值应该是A，如果V的确是A，就将V更新为B；但如果预期错误，V不为A，则不更新V，无论如何，结束后将V的旧值返回给我”。就是这么简单的逻辑，构筑了并发包的基础。

硬件支持

CAS之所以如此有效是因为它是一个原子操作，现在的CPU架构均支持CAS原子操作，在x86中，这个CPU指令是cmpxchg指令，这可以在阅读JNI调用的C++源码中看到（Atomic::cmpxchg）：

有了硬件支持，CAS操作就可以大施拳脚了。

背景

之前文章里介绍的是普通的二叉树非递归遍历方案，这些方案最终都需要基于栈来实现，这是因为我们只能获取到当前节点到子节点的引用，无法获取到上层节点的引用，只能通过栈把它们的信息暂时保存下来了。
那么能不能不依托于栈，就能记录上层节点的信息，同时又保证可以正确回溯呢？通过左神的书，学习到了这种牛x的遍历方法——Morris遍历方法。

描述

二叉树的各序遍历非递归实现——O(h)空间复杂度

发表于 2018-01-22 | 分类于 数据结构与算法

背景

对于二叉树的三种遍历，用递归当然是比较的简单直接了，而递归实质上是程序利用方法栈不断替换当前栈帧来实现的。
因此我们也可以用自己设计的数据结构来替代方法栈，用循环来代替递归，比较简单的就是直接用栈结构来实现。
本文介绍的方法虽能解决问题，但是与递归一样，需要用到O(h)的空间复杂度，h为树高，之后将会介绍一种不依赖于栈的二叉树遍历方法，其空间复杂度仅为O(1)。

核心

如何根据前序、中序、后序遍历的不同要求(分裂和执行的顺序等)来设计入栈和出栈的功能。本文中的二叉树结构如下所示（按leetcode的标准）。

前序遍历

在前序遍历中，每个节点的执行在分裂之前进行。实现时，将栈顶节点出栈，完成执行后再并将非空的子节点放入栈中即可。

中序遍历

中序遍历比前序遍历要复杂一些，因为中序遍历中，节点的执行发生在左子节点分裂和右子节点分裂中间。
解决方案是，在栈中仍然仅存放非空节点，并增加一个节点引用来辅助解决问题。（从左神的书学习。。）。

后序遍历

后序遍历中，节点的执行发生在左子节点分裂和右子节点分裂之后，因此可以将分裂和执行这两个过程分离开来，用两个栈来实现，所有的执行工作在第二个栈中进行。

Java与JVM：（三）关于引用传递的文字游戏——从一道题目想到

做某客的Java题时遇到一道题目，一时间有点懵逼，毕竟值传递啊引用传递啊这种有点类似文字游戏的概念被用来出题真的好么，特别是在Java语言下。



答案是A、C、D，也是有点懵，毕竟就传统意义来讲，Java应该只有值传递，没有引用传递吧。

Java有没有引用传递？

个人感觉这种就是文字游戏了，当然就大多数定义来讲，以及与其他语言比较的话（C++， C#），Java的确是没有引用传递，只有值传递的。

一般来说，值传递是指在调用函数时，传递的参数是实参值的一份拷贝,引用传递则是指在调用函数时，传递的参数是实参的真实地址

相信写过C#就对这两个的区别很清晰了，在C#中用ref、out关键词可以使用引用传递，这也可以看作对C++的一种继承。而在Java中，即使参数是对象类型数据，在调用时，也仅仅是传递了引用（reference）的拷贝而已。这也是为什么在以下代码中，str的值无法改变，而在C#中则可以。

C#:

如果非得说Java有引用传递呢？

如果非得说Java有引用传递，那么只能按值的类型来划分了，把基本类型参数的传递看作是值传递，把引用类型（对象）参数类型的传递看作是引用传递了，这样的话，这道题目的解的确是A、C、D。

这世上只有值传递？

如果硬把所有传递的是”值“的传递就作为”值传递“，那么在C#和C++中所谓的”引用传递“也不过是把真实的引用地址这个”值“传过去了，要这么咬文嚼字的话，那世上也就没有”引用传递“了。

总结

不管是中文还是英文，一个单词的概念也不是一成不变的，有时候还是得分场景来说，纠结于Java有没有引用传递，很多时候有点像一个文字游戏。当然，这个游戏在大部分情况是有答案的——那就是在Java中，没有引用传递，只有值传递。

Java与JVM：（二）从逃逸分析到栈上分配

发表于 2017-12-31 | 分类于 Java

背景

传统概念中Java的对象均保存在堆中，被所有线程共享，由垃圾回收机制对其进行回收。这样做的优点在于可以对所有对象进行统一而清晰的管理，缺点在于增加了垃圾回收器的压力。由于大量的对象均只存在于某一方法或某一线程中，因为如果将它们和本地基本变量一样随着栈帧出栈而销毁，垃圾回收器就可以省心不少。

逃逸分析

书接上文，如果能确定对象仅存在于某方法或某线程中，就可以进行优化，所以关键的一步在于判断某对象是否在声明该方法的方法或线程之外被调用了，当项目很大时，这一步还是要花很多的时间。逃逸分析对对象的动态作用域进行分析，包括方法逃逸和线程逃逸，它们的区别见名即知义。

栈上分配

栈帧上存储的原始类型变量（包括基本类型变量以及引用）都会随着栈帧的出栈操作而自动销毁，无需垃圾回收机制对它们进行回收，因此如果逃逸分析能够确定对象不会发生逃逸，则可以在栈上分配内存。

标量替换

直接的栈上分配看起来很美，但是实现起来还是非常难的，因此Java目前采用标量替换的方式来实现栈上分配。既然在栈帧上分配和销毁原始类型变量是容易的，所以可以将对象分解成原始类型变量存储，以它们的组合来表征该对象。
